## 2. データ共有アーキテクチャ

### 2-1. 共有ファイルと参照パス

| ファイル | WEB版 | Mobile版 |
|----------|-------|----------|
| `japan_history_data.json` | `'japan_history_data.json'`（同階層） | `'../japan_history_data.json'`（一つ上） |
| `japan.topojson` | `'japan.topojson'`（CDN フォールバックあり） | `'../japan.topojson'`（FB なし） |

**両版が同一ファイルを読む。JSON を 1 度コミットすれば両方に反映される**。
mobile デプロイは `mobile/sw.js` の `CACHE` 値を `jhm-vNN` で +1 する必要あり（後述）。

### 2-2. WEB版のデータ読込フロー（3 段階優先順位）

```
① インライン最小 seed（index.html 内ハードコード — 非常時フォールバック）
       ↓ 上書き
② fetch('japan_history_data.json', {cache:'no-store'})
   ← サーバー最新を毎回取得（JSON が正のマスター）
       ↓ マージ（version 比較）
③ localStorage（ユーザーの未保存編集）
   ← JSON.version > LS.version の場合は LS を破棄し JSON を優先
```

主要関数（index.html）:
- `loadData()` — 上記 3 段階を実装
- `saveData()` — 編集のたびに localStorage に保存
- `exportData()` — `version+1` で JSON ダウンロード → リポジトリに差し替えてコミット
- `importData()` — JSON ファイルを読み込んで全データ上書き
- `DATA_VERSION_FALLBACK = 18` — JSON 取得失敗時の既定 version

### 2-3. Mobile版のデータ読込

Service Worker による **ネットワークファースト**（`mobile/sw.js`）:
```
fetch → 成功: キャッシュ更新して返す
       → 失敗: caches.match() でオフライン応答
```
バージョン更新時は `CACHE = 'jhm-vNN'` を +1 する。古いキャッシュは `activate` イベントで一括削除される。

## 3. データモデル（`japan_history_data.json`）

### 3-1. ルート構造

```json
{
  "version": 18,
  "persons":    [14 件],   // 一般人物（実朝・北条政子 など）
  "emperors":   [27 件],   // 天皇（神武〜明仁）
  "rulers":     [17 件],   // 為政者（藤原道長・徳川家康 など）
  "religious":  [12 件],   // 宗教者（最澄・空海・親鸞 など）
  "pms":        [66 件],   // 総理大臣（伊藤博文〜現代）
  "rakugo":     [51 件],   // 落語家
  "kodan":      [20 件],   // 講談師
  "noh":        [39 件],   // 能曲（演目）
  "nohActs":    [5 段],    // 五番立解説（神/男/女/狂/鬼）
  "cats":       [3 カテゴリ / 計 57 項目]  // 宗教・芸能・出来事
}
```

### 3-2. 人物系（emperor/ruler/religious/pm/rakugo/kodan/person）

```json
{
  "id":   "iemitsu",
  "name": "徳川家光",
  "fg":   "とくがわ いえみつ",        // ふりがな（検索用）
  "s":    1604,                      // 生年（start）
  "e":    1651,                      // 没年（end / 不明は 9999）
  "era":  "edo",                     // 主活躍時代 ID（ERAS 参照）
  "desc": "江戸幕府 3 代将軍。…",
  "locs": [                           // 地図ピン
    {"r":"kanto",     "tp":"birth",  "lb":"江戸城（誕生）"},
    {"r":"kinki",     "tp":"active", "lb":"上洛"},
    {"r":"kanto",     "tp":"active", "lb":"江戸城"}
  ]
}
```

**`tp`（種別）と色対応** — レンダリング時に `LOCTPCOLOR` で強制統一:
| tp 値 | 色 | 意味 |
|-------|-----|------|
| `birth` | `#50cc50` 緑 | 誕生地 |
| `active` | `#6088ff` 青 | 活躍地 |
| `exile` | `#ff8050` 橙 | 配流地 |

### 3-3. 時代（ERAS / フロント側 const）

```js
[
  {id:'jomon',  s:-10000, e:-300,  c:'#8b6f47', n:'縄文'},
  {id:'yayoi',  s:-300,   e:300,   c:'#a89060', n:'弥生'},
  ...
  {id:'reiwa',  s:2019,   e:9999,  c:'#777',   n:'令和'}
]
```
`eraForYear(y)` で年→時代を判定。`cats` の項目色は `eraForYear(item.s).c` で**時代色に強制統一**（個別色入力欄なし）。

### 3-4. cats（カテゴリ項目: 宗教 / 芸能 / 出来事）

```json
{
  "id": "religion",
  "title": "🛐 宗教",
  "items": [
    {"id":"tendai", "n":"天台宗", "fg":"てんだいしゅう", "s":806, "e":9999, "desc":"…"}
  ]
}
```
- 各項目に `fg`（ふりがな）— 検索のため全項目登録済み
- `e=9999` は「存続中」を意味

### 3-5. noh（能曲）

```json
{
  "id":"hachinoki",
  "name":"鉢木（はちのき）",
  "fg":"はちのき",
  "act":4,                                 // 1〜5（五番立）
  "category":"現在物（人情物・武家物）",
  "shite":"佐野源左衛門尉常世",
  "waki":"北条時頼",
  "source":"鉢木伝説",
  "era":"室町前期",
  "author":"世阿弥（伝）",
  "religion":"仏教（禅宗）",
  "note":"舞台：上野国佐野…"
}
```
`nohActs` は 1〜5 の `{title, summary, color}` 配列。

### 3-6. ユーザー記録（localStorage / JSON 化はしない）

| キー | 用途 |
|------|------|
| `nenpyo_data_v1` (WEB) | 編集中の全データ |
| `noh_records_v1` (両版共通) | 能の鑑賞記録 `{曲ID:{watched,place,date,memo}}` |

## 6. 鑑賞記録（noh records）と Safari 履歴削除対策

### 6-1. 仕様

| 項目 | 内容 |
|------|------|
| キー | `noh_records_v1`（WEB / Mobile 共通） |
| 構造 | `{ "曲ID": {"watched":bool, "place":str, "date":"YYYY-MM-DD", "memo":str} }` |
| 保存タイミング | フォーム入力のたびに `saveNohRecord(id, rec)` |
| 互換 | WEB と Mobile で同じキー・同じ構造 → 端末間移動可 |

### 6-2. Safari の履歴削除問題（仕様であり、コードで防げない）

Safari (macOS / iOS) の **「履歴と Web サイトデータを消去」は localStorage も含む全サイトデータを消す**。これはプライバシー保護の仕様。IndexedDB / Cookie / Cache に変えても同じ運命。
**コミット bdca08e** で書出 / 読込機能を実装した。

### 6-3. `exportNohRecords()` / `importNohRecords(input)`

書出ファイル形式:
```json
{
  "key": "noh_records_v1",
  "version": 1,
  "exportedAt": "2026-06-27T01:23:45.678Z",
  "records": {
    "hachinoki": {"watched":true, "place":"…", "date":"2026-06-20", "memo":"…"},
    "takasago":  {"watched":true, ...}
  }
}
```
- ファイル名: `noh_records_YYYY-MM-DD.json`
- **import は merge**（既存に上書きせず追加）、新規/上書き件数を alert で通知
- 不正 JSON / `key` 不一致は alert で拒否

### 6-4. UI 配置

| 版 | 配置 |
|----|------|
| WEB | `#noh-list-drawer` ヘッダ（タイトル横）に `💾 記録書出` / `📂 記録読込` |
| Mobile | `screen-noh-list` ナビバー右端に `💾 書出` / `📂 読込`（短縮表記） |

ボタン色: 薄緑系（`#0c1a10` 背景 / `#a8d8b8` 文字 / `#4a6a4a` ボーダー）

## 7. 地図ピン描画ルール

### 7-1. `LOCTPCOLOR`（種別→色 のレンダリング時上書き）

```js
var LOCTPCOLOR={birth:'#50cc50', active:'#6088ff', exile:'#ff8050'};
```
JSON 内の `loc.c` を信用せず、レンダ時に `LOCTPCOLOR[loc.tp]||loc.c` で上書き。
**「活躍は常に青」を強制する** ための仕組み。

### 7-2. 沖縄領域（Mobile版 / 重要）

地図 3 倍ズーム時も**沖縄領域は画面固定**。clip-path の座標解釈に注意:
- `userSpaceOnUse` の clipPath は**親の transform 適用後の local 座標**
- 親の `scale(3)` 後でも rect 座標は変えてはいけない（固定値 `(2,2,140,140)` のまま）
- 初期実装で外側座標系で計算してしまい誤動作 → 修正済み（commit 577562f）

## 10. 既知の罠・注意点（やってしまいがちな失敗）

### 10-1. データ系

- **新規人物追加で `fg` を忘れる** → 検索でヒットしない。**必須**。
- **`tp` を `JSON` 内で雑に塗り分け** → どうせ `LOCTPCOLOR` で上書きされる。`loc.c` は書いてもよいが**機能しない**。
- **`e=9999` の人物**（現存）に `era` を入れ忘れる → 時代抽出が `ERAS[0]` にフォールバックする。
- **天皇.txt のような外部入力ファイルから流し込むとき** 改行欠落で `l` フィールドに概要が混入するケースあり（実例: 宇多天皇 / commit f43fa59）。
- **DASH_CHARS（：—／—）で終わる名前** が外部入力に混じることがある（落語家 9 件）。`fg` 計算前に末尾の `：—` を strip すべし。

### 10-2. UI 系

- **`#lp` から `position:relative` を外すと** `#noh-list-drawer` が viewport 全体に広がる。
- **能の鑑賞メモの textarea**: 画面が hidden の状態だと `scrollHeight===0`。`offsetParent===null` でチェックし、表示直後に `setTimeout` で再計算。
- **WEB と Mobile のタイトル / アイコン**は **必ず両方を同時に直す**。片方だけだと「同じコンテンツなのに名前違う」状態になる（commit 079f940 が教訓）。

### 10-3. デプロイ系

- **`mobile/sw.js` の `CACHE` バージョンを上げ忘れる** が一番事故りやすい。データ変えたら必ず +1。
- **`DATA_VERSION_FALLBACK`（index.html 内）と JSON の version** を合わせる。

### 10-4. Safari 系

- **localStorage は Safari の「履歴削除」で消える**。`exportNohRecords` でユーザー自身に定期バックアップしてもらう設計（既に実装済み）。
- iCloud Drive / Dropbox 等への退避を案内するのが現実解。
